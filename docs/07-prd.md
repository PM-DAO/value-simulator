# Product Requirements Document: Value Simulator

## 1. プロダクトビジョン

**「サービスアイデアを入力するだけで、そのサービスが市場でどう広がるかが見える」**

起業家・プロダクトマネージャーが新規サービスのアイデア段階で、市場浸透のシミュレーションを回せるWebアプリケーション。日本市場の1/1000スケール仮想空間（100エージェント）上で、Bass拡散理論に基づくエージェントベースシミュレーションを実行し、採用曲線・売上予測・改善ポイントを可視化する。

## 2. ターゲットユーザー

| ペルソナ | ニーズ |
|---------|-------|
| **起業家**（アイデア検証段階） | 「このアイデア、実際に人が使うのか？」を数字で確認したい |
| **PM**（新機能企画） | 価格戦略やターゲット変更が採用にどう影響するか事前に検証したい |
| **投資家/アクセラレータ** | ピッチされたサービスの市場浸透ポテンシャルを定量評価したい |

## 3. ユーザーストーリー

### 必須（Phase 1で実現）
- ユーザーとして、サービス名と価格を入力すると、90日間の採用曲線（S字カーブ）がグラフで見える
- ユーザーとして、同じ入力でも毎回微妙に異なる結果が出る（確率的シミュレーション）ことで、リアリティを感じられる

### 重要（Phase 2で実現）
- ユーザーとして、サービスの説明・ターゲット・カテゴリ・価格を入力すると、それに応じてシミュレーション結果が変わる
- ユーザーとして、価格を変えたときに採用率がどう変わるかを確認できる
- ユーザーとして、Rogers普及カテゴリ別（イノベーター〜ラガード）の採用内訳が見える
- ユーザーとして、累積ユーザー数と売上推移のグラフが見える

### 理想（Phase 3で実現）
- ユーザーとして、サービスの説明文だけ入力すれば、AIがJTBD・ターゲット・パラメータを自動推論してくれる
- ユーザーとして、AIが推論したパラメータを手動で微調整できる
- ユーザーとして、「広告あり vs なし」「価格A vs B」など複数シナリオを重ねて比較できる
- ユーザーとして、マーケティング施策のタイムラインを設定し、その効果をシミュレーションできる

## Phase 1: 動くプロトタイプ — 「S字カーブが見える」

**Day 1〜2 / ゴール: エージェントベースシミュレーションのE2Eが動く**

### 1.1 機能要件

#### 入力画面

| フィールド | 型 | 必須 | 説明 |
|-----------|-----|------|------|
| サービス名 | text | Yes | 表示用ラベル |
| 想定価格（月額/円） | number | Yes | 0〜100,000 |
| 想定市場規模 | select | Yes | 小（1万人）/ 中（10万人）/ 大（100万人） |

- 入力フォームは1画面。送信ボタン1つ
- バリデーション: 価格は正の整数、サービス名は1〜100文字

#### シミュレーションエンジン（Phase 1）

**Day 1: 数式ベース（ハリボテ）**
- Bass拡散モデルの閉形式解: `F(t) = (1 - e^(-(p+q)t)) / (1 + (q/p)e^(-(p+q)t))`
- パラメータ: p=0.03（イノベーション係数）、q=0.38（模倣係数）
- 入力の価格から市場規模を調整（高価格→到達率低下）
- 出力: 90日分の日次新規採用者数 + 累積採用者数

**Day 2: エージェントベース**
- 100体のエージェントを生成（最小属性: 年齢、所得、行動タイプ）
- Watts-Strogatz小世界ネットワーク（k=6, p=0.3）で接続
- 各タイムステップで非採用エージェントが `P(adopt) = p + q × (採用済み隣接/全隣接)` で判定
- Rogersカテゴリ（イノベーター2.5%、アーリーアダプター13.5%、...）に基づくp/q個別化

#### 結果画面（Phase 1）

| コンポーネント | 表示内容 |
|--------------|---------|
| **採用曲線チャート** | X軸: 日数(0-90), Y軸: 累積採用者数。S字カーブ |
| **日次新規採用チャート** | X軸: 日数, Y軸: 新規採用者数。ベル型カーブ |
| **サマリー数値** | 90日後の総採用者数、最大日次採用数、採用率 |

- チャートライブラリ: Recharts
- レスポンシブ対応は不要（デスクトップのみ）

#### API仕様（Phase 1）

```

Request:
{
  "service_name": "string",
  "price": number,
  "market_size": "small" | "medium" | "large"
}

Response:
{
  "service_name": "string",
  "config": {
    "num_agents": number,
    "num_steps": number,
    "p": number,
    "q": number
  },
  "daily_adoption": [number],     // 90要素
  "cumulative_adoption": [number], // 90要素
  "summary": {
    "total_adopters": number,
    "peak_daily": number,
    "adoption_rate": number        // 0.0-1.0

}
```

## Phase 2: 賢いシミュレーション — 「パラメータで結果が変わる」

**Day 3〜4 / ゴール: サービス特性がシミュレーションに反映され、リッチなダッシュボードで可視化**

### 2.1 機能要件

#### 入力画面（拡張）

Phase 1の入力に加えて:

| フィールド | 型 | 必須 | 説明 |
|-----------|-----|------|------|
| サービス説明 | textarea | No | 自由記述（最大500文字） |
| ターゲットユーザー | select | No | 学生 / 若手社会人 / ファミリー / シニア / ビジネス |
| カテゴリ | select | Yes | SaaS / EC / メディア / フード / モビリティ / ヘルスケア / 教育 / エンタメ / 金融 |
| 価格モデル | select | Yes | 無料 / フリーミアム / サブスク / 従量課金 / 買い切り |
| 競合の存在 | select | Yes | なし / 弱い / 強い |

#### JTBD / 市場エンジン

**Jobs-to-be-Done (JTBD)**
- カテゴリ×ターゲットの組み合わせからプリセットJobを割り当て
- 各Jobに `importance`（1-5）と `current_satisfaction`（1-5）を設定
- ODI機会スコア = `importance + max(importance - satisfaction, 0)`
- スコア > 10: 未充足（高機会）、6-10: 適切、< 6: 過剰充足

**価格感度（WTP: Willingness to Pay）**
- 各エージェントのWTPを対数正規分布から生成: `WTP ~ LogNormal(μ, σ)`
- μ, σ はエージェントの所得と行動タイプから決定
- `価格 > WTP` のエージェントは採用しない

**統合採用判定**
各タイムステップで非採用エージェント i の採用確率:
```
P_adopt(i) = P_bass(i) × JTBD_fit(i) × price_ok(i)

P_bass(i)     = p_i + q_i × (採用済み隣接 / 全隣接)
JTBD_fit(i)   = ODI_score / 15  （0.0〜1.0に正規化）
price_ok(i)   = 1 if price ≤ WTP(i) else 0
```

**競合影響**
- 競合「なし」: 基本パラメータそのまま
- 競合「弱い」: 到達可能市場を80%に縮小
- 競合「強い」: 到達可能市場を50%に縮小 + qを20%低下

#### 人口統計（日本簡易版）

100体のエージェントを以下の分布で生成（e-Stat 2020国勢調査ベース）:

| 属性 | 分布 |
|------|------|
| 年齢 | 20代 18%, 30代 16%, 40代 18%, 50代 16%, 60代 15%, 70代+ 17% |
| 性別 | 男性 49%, 女性 51% |
| 所得 | 〜300万 30%, 300-500万 30%, 500-700万 20%, 700万+ 20% |
| 地域 | 関東 35%, 関西 15%, 中部 12%, 九州 10%, その他 28% |

Rogersカテゴリは行動タイプとして割当:
| カテゴリ | 割合 | p倍率 | q倍率 | 価格感度 |
|---------|------|-------|-------|---------|
| イノベーター | 2.5% | 3.0x | 0.5x | 低 |
| アーリーアダプター | 13.5% | 2.0x | 0.8x | 低〜中 |
| アーリーマジョリティ | 34% | 1.0x | 1.2x | 中 |
| レイトマジョリティ | 34% | 0.5x | 1.5x | 高 |
| ラガード | 16% | 0.2x | 0.3x | 非常に高 |

Phase 1に加えて:

| **ODI機会スコア** | スコア値 + 「未充足 / 適切 / 過剰充足」ラベル |
| **エージェント分布** | 年齢×所得のバブルチャート。採用/未採用を色分け |
| **サマリーカード** | 90日後売上予測、採用率、平均採用日数 |

## Phase 3: AIパワー + 意思決定ツール — 「全自動シミュレーション + シナリオ比較」

**Day 5〜7 / ゴール: 説明文入力で全自動、複数シナリオ比較、プロダクト品質**

### 3.1 機能要件

#### 入力モード切り替え

**簡易モード（AI推論）**
- 入力: サービス説明文（テキストエリア、最大1000文字）のみ
- Claude APIがサービス説明文から以下を自動推論:
  - JTBD（functional / emotional / social）
  - ターゲットユーザー
  - カテゴリ
  - 推奨価格帯
  - 競合状況
- 推論結果をプレビュー表示し、ユーザーが確認・修正できる
- 確認後にシミュレーション実行

**詳細モード（手動設定）**
- Phase 2と同じフォーム。パワーユーザー向け

#### Claude API JTBD推論仕様

```
入力: サービス説明文
出力:
{
  "jobs": [
    {
      "statement": "string",         // e.g. "通勤時間を有効活用する"
      "type": "functional" | "emotional" | "social",
      "importance": 1-5,
      "current_satisfaction": 1-5
    }
  ],
  "target": "string",
  "category": "string",
  "suggested_price": number,
  "competition": "none" | "weak" | "strong",
  "reasoning": "string"             // 推論理由（ユーザー向け説明）
}
```

- Fallback: Claude API障害時は詳細モードにフォールバック
- コスト: 1リクエストあたり ~$0.01（Haiku使用想定）

#### シナリオ比較

- 最大3シナリオを同時実行・比較
- ユーザーがシナリオごとにパラメータを変更できる:
  - 価格差分
  - マーケティング施策の有無
  - ターゲット変更

**マーケティング施策モデル**

| 施策 | パラメータ効果 |
|------|--------------|
| PR/プレスリリース | Day Xに1回限りの `p` ブースト（3x、1日間） |
| SNS広告 | 指定期間中 `p` を1.5xに継続ブースト |
| インフルエンサー | Day Xに特定エージェント（オピニオンリーダー）を強制採用 |
| 口コミキャンペーン | 指定期間中 `q` を1.3xにブースト |

ユーザーはタイムライン上に施策を配置:
```
Day 1                Day 30              Day 60              Day 90
|----PR発表----|------SNS広告------|-----口コミCP------|
```

#### 結果画面（Phase 3拡張）

Phase 2に加えて:

| コンポーネント | 表示内容 |
|--------------|---------|
| **JTBD推論プレビュー** | AIが推論したJob一覧、編集可能 |
| **シナリオ比較チャート** | 複数シナリオのS字カーブを重ね描画（色分け） |
| **シナリオ比較テーブル** | 90日後の採用者数、売上、採用率の横並び比較 |
| **改善提案** | ODI分析に基づく改善サジェスチョン（テキスト） |
| **マーケティングROI** | 各施策の投入コスト vs 獲得ユーザー数 |

## 6. 技術的制約と前提

| 制約 | 対応 |
|------|------|
| エージェント100体は統計的に不十分 | MVP後に1000→10000にスケールアップ予定。Phase 1-3では定性的な傾向の確認に価値がある |
| Bass拡散は簡略化モデル | 精度より「パラメータ変更→結果変化」の因果関係の直感的理解に重点 |
| 日本人口統計は簡易版 | e-Statの公開集計データに基づく近似。IPF合成はMVP後 |
| Claude APIコスト | Phase 3のみ使用。1回~$0.01。Fallbackあり |
| シングルユーザー前提 | MVP。マルチユーザー対応はMVP後 |
