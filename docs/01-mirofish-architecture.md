# MiroFishのアーキテクチャ全体像

## 概要

MiroFishは、北京の学部生・郭杭江が約10日間で開発したプロジェクトで、Shanda Groupから約41億円の出資を受けた。GitHub上で40.8K starsを獲得し、AGPL-3.0ライセンスで公開されている。

## 技術スタック

| レイヤー | 技術 |
| --- | --- |
| フロントエンド | Vue 3 + Vite + D3.js v7（グラフ可視化） |
| バックエンド | Python 3.11+ / Flask 3.0+ |
| シミュレーション | OASIS v0.2.5 + CAMEL-AI v0.2.78 |
| 知識グラフ | Zep Cloud SDK 3.13.0（フォーク版ではNeo4jやKuzuDB） |

## 五段階パイプライン

1. **文書アップロード** → LLMによるオントロジー抽出（10エンティティ型、10関係型）
2. **知識グラフ構築** → Zep Cloud GraphRAGによるエンティティ・関係の抽出
3. **エージェントペルソナ生成** → LLMが知識グラフのエンティティから人口統計的に多様なペルソナを自動生成
4. **二重プラットフォームシミュレーション** → Twitter風（ウイルス拡散・インフルエンサー）+ Reddit風（スレッド議論・コミュニティ）を並列実行
5. **レポート生成** → ReACTパターンによるエージェントインタビュー付き分析レポート

バックエンドは40以上のRESTエンドポイントを5つのサービス群（グラフ構築、エンティティ読取、設定生成、シミュレーション実行、レポート生成）に分散して提供する。

## エージェント設計

各エージェントは以下の属性を持つ独立AIエンティティとして設計されている：

| 属性 | 説明 | 値の例 |
| --- | --- | --- |
| アイデンティティ | 名前、ユーザー名、プロフィール | @ZhangWei_Student |
| ペルソナ | 性格、バックグラウンド、意見 | AI倫理を研究する大学院生 |
| 活動レベル | 投稿頻度（0.0-1.0） | 0.8 = 非常にアクティブ |
| アクティブ時間 | オンライン時間帯（配列） | [8,9,10,11,18,19,20,21,22,23] |
| 影響力重み | コンテンツ可視性の乗数 | 0.8（低）〜3.0（高） |
| スタンス | トピックへの立場 | 支持、反対、中立、傍観者 |

意思決定ロジックはOASISフレームワークに委任される。エージェントはアクティビティスケジュールに基づいて起動し、レコメンデーションシステム（TwHIN-BERTを使用した興味ベース＋ホットスコアベースアルゴリズム）からコンテンツを受信し、LLMの「脳」がペルソナ＋記憶＋推薦コンテンツのコンテキストに基づいて23種類のソーシャルアクション（投稿、コメント、リポスト、いいね、フォローなど）のうちどれを実行するかを決定する。

## メモリ・状態管理

メモリアーキテクチャは三層構造で設計されている：

- **個別メモリ** — エージェントごとのアクション履歴と信念状態
- **集団メモリ** — グラフ構築時に注入される共有グループ情報
- **エージェント分離** — run_idによるメモリ交差汚染防止

### 定期的要約によるコスト最適化

NサイクルごとにLLMが長期履歴を核心的信念、関係性、確信度の変化に凝縮する（約500トークンの要約＋直近20ターン）。これにより、完全な履歴再生と比較してトークンオーバーヘッドを大幅に削減する。

### メモリが生み出すヒステリシス効果（経路依存性）

- **偏極波** — 初期の対立が固定化された立場に硬化する
- **確信シフト** — 繰り返しの反証に曝されたエージェントが信念を漸次更新する
- **派閥凝集** — 共有された記憶が創発的グループアイデンティティを形成する

## 時系列シミュレーション

OASISの時間エンジンが1タイムステップ＝3分のシミュレーション時間を提供する。典型的なシミュレーションは72時間のシミュレーション時間で、両プラットフォームにわたって数千のアクションを生成する。

## スケーリングとLLM統合

OASISエンジンは最大100万エージェントのシミュレーションをサポートする。5台のA100 GPU（1台はRecSys、4台はvLLMによるLLM推論）を使用して、10万ユーザー×10タイムステップを約2日で処理するベンチマークが報告されている。ただしMiroFish単体の実用範囲は50〜200エージェントで、500以上はより多くの計算リソースが必要となる。

### LLM統合ポイント（6箇所）

1. オントロジー生成
2. ペルソナ生成
3. シミュレーション設定生成
4. シミュレーション中のエージェント推論
5. レポート生成
6. 事後インタビュー

任意のOpenAI SDK互換エンドポイント（Qwen-plus、GPT-4、Claude、ローカルOllama等）を利用可能。
