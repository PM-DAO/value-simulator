# 7日間アジャイルスプリント計画

## 原則

**毎日動くものがある。毎日触れる。毎日デモできる。**

Day 1で最もシンプルなE2Eが動き、Day 2以降は毎日機能を積み重ねて進化させる。

---

## Day 1: 「ハリボテE2E」— 入力→シミュレーション→結果表示が動く

### ゴール
ブラウザでサービス名を入力 → バックエンドでダミーシミュレーション → S字カーブがグラフ表示される

### やること
1. **Backend**: Python/FastAPI セットアップ + `/api/simulate` エンドポイント
   - 入力: サービス名、価格
   - 処理: ハードコードされたBass拡散（純粋数式、エージェントなし）
   - 出力: 90日分の採用者数配列
2. **Frontend**: Next.js セットアップ + 入力フォーム + Rechartsでグラフ描画
3. **接続**: フロント→API→結果表示のE2Eが動く

### 触れるもの
- ブラウザで `localhost:3000` を開き、サービス名を入力→S字カーブが表示される

### テスト
- `test_api.py`: POST /api/simulate → 200 + 配列が返る
- `test_diffusion.py`: Bass数式が正しいS字カーブを返す

---

## Day 2: エージェントモデル導入 — ダミー→本物のエージェントに置換

### ゴール
同じUI、でも裏側が10体のエージェントベースシミュレーションに進化

### やること
1. **agent.py**: BDIエージェントモデル（Pydantic）— 最小限の属性（年齢、所得、行動タイプ）
2. **population.py**: 10体のエージェント生成（ハードコード分布でOK）
3. **network.py**: NetworkXで10ノードの小世界グラフ
4. **diffusion.py**: エージェントベースBass拡散（ネットワーク上）
5. **engine.py**: 統合エンジン（population → network → diffusion → 結果集計）
6. **api.py更新**: ダミー数式→エージェントエンジンに差し替え

### 触れるもの
- 同じUIだが、結果が毎回微妙に異なる（確率的エージェントベース）
- エージェント数を10→50→100と変えてS字カーブの変化を確認

### テスト
- `test_agent.py`: エージェント生成・属性検証
- `test_population.py`: N体生成・分布チェック
- `test_network.py`: 小世界特性（クラスタ係数、パス長）
- `test_diffusion.py`: エージェントベース拡散がS字を描く
- `test_engine.py`: E2E統合テスト

---

## Day 3: JTBD + 入力を豊かに — サービス特性がシミュレーションに影響

### ゴール
サービスの説明・ターゲット・カテゴリを入力すると、それに応じてシミュレーション結果が変わる

### やること
1. **jtbd.py**: Job/Outcomeモデル、ODI機会スコア計算
2. **market.py**: 価格感度（WTP対数正規分布）、効用計算
3. **engine.py更新**: JTBD充足度 × 価格許容 × ソーシャル影響 の統合採用判定
4. **フロント更新**: 入力フォーム拡張（説明、ターゲットユーザー、カテゴリ、価格帯）
5. **API更新**: 拡張入力をエンジンに渡す

### 触れるもの
- 価格を変えるとS字カーブの高さが変わる
- ターゲットを変えると採用速度が変わる
- 「高価格×ニッチ」vs「低価格×マス」の比較ができる

### テスト
- `test_jtbd.py`: ODI機会スコア計算
- `test_market.py`: WTP分布、効用計算
- `test_engine.py`: 入力パラメータ変更→結果変化の検証

## Day 5: Claude API JTBD推論 — サービス説明だけで全自動シミュレーション

### ゴール
サービスのアイデア文を書くだけで、JTBD・ターゲット・パラメータが自動推論され、シミュレーションが走る

### やること
1. **jtbd.py拡張**: Claude APIでサービス説明文→JTBD/ターゲットユーザー/カテゴリ自動推論
2. **API更新**: 「簡易モード」（説明文だけ入力）と「詳細モード」（手動パラメータ設定）の2モード
3. **フロント更新**: 簡易入力モードUI + 推論結果のプレビュー表示（ユーザーが確認・修正できる）
4. **結果画面追加**: ODI機会スコア表示、改善提案テキスト

### 触れるもの
- 「Uberのような配車サービス」と入力→自動で全パラメータ設定→シミュレーション実行
- 推論されたJTBDを手動で微調整→結果の変化を確認

### テスト
- Claude API統合テスト（モック + 実API）
- Fallback: LLMなしでも手動入力で動作することを確認

## アーキテクチャ概要

```
[Browser - Next.js]
    ↓ POST /api/simulate
[FastAPI Backend]
    ├── Claude API → JTBD推論 (Day 5~)
    ├── PopulationGenerator → N agents
    ├── NetworkBuilder → Watts-Strogatz graph
    ├── SimulationEngine
    │   ├── Bass Diffusion (per step)
    │   ├── Market Engine (price/utility) (Day 3~)
    │   ├── JTBD Matching (Day 3~)
    │   └── Marketing Injection (Day 6~)
    └── Results → JSON response
[Browser]
    └── Recharts → 採用曲線・売上チャート
```

## リスクと対策

| リスク | 対策 |
|--------|------|
| Day 1のE2Eが動かない | フロント・バックを最小限に。CORSだけ先に解決 |
| LLM APIコスト | Day 5まではLLM不要。Day 5もfallbackあり |
| エージェント数増で遅い | Day 2は10体から。パフォーマンス問題が出てから最適化 |
| フロント工数超過 | Day 1は最小UI。毎日少しずつ拡張 |
